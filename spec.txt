Contexto:
Estamos criando uma base de padrões versionada para humanos e agentes de IA (Cursor, Claude Code, Devin e outros), aplicável a qualquer projeto de engenharia de dados e integrações na AWS.

* Evite documentação genérica.
* Traga exemplos concretos.
* Escreva em português do Brasil.

Regras para `AGENTS.md`:

* Deve ser o documento central e interoperável para agentes de IA.
* Deve conter:

  * contexto do projeto;
  * objetivo dos padrões;
  * princípios inegociáveis;
  * comandos de build/test/lint, se existirem;
  * padrão de resposta esperado dos agentes;
  * quando alterar código e quando apenas propor;
  * como lidar com dúvidas;
  * regra de não inventar comportamento de negócio;
  * regra de sempre procurar documentação do projeto antes de implementar;
  * regra de sempre criar/atualizar testes;
  * regra de sempre avaliar observabilidade e performance;
  * regra de preservar compatibilidade com padrões existentes;
  * regra de registrar decisões arquiteturais relevantes em ADR.

Regras para `CLAUDE.md`:

* Deve importar ou resumir o `AGENTS.md`.
* Deve ser curto e objetivo.
* Deve orientar Claude Code a usar as Skills quando a tarefa for procedimento repetível.
* Deve indicar que regras grandes ficam em `.claude/rules`.
* Deve evitar conteúdo excessivo que polua o contexto.
* Deve conter instruções específicas para:

  * sempre planejar antes de editar;
  * sempre rodar testes quando possível;
  * sempre explicar impactos;
  * nunca alterar contratos públicos sem destacar breaking change;
  * nunca remover observabilidade;
  * nunca reduzir cobertura de testes.

Regras para `.claude/rules`:

* Cada arquivo deve ser focado por tema/stack.
* Cada regra deve ser objetiva e acionável.
* Use bullets, “faça/não faça”, exemplos e critérios de aceite.
* Evite duplicação excessiva com os documentos humanos.
* As rules devem funcionar como lembretes operacionais para agentes.

## 10. TaaC — testes automatizados integrados na pipeline

Crie `docs/padroes/09-taac-testes-integrados-pipeline.md`.

Definição:
TaaC aqui significa testes automatizados acima dos testes unitários, executados na pipeline, validando integração realista entre componentes com ambiente autocontido sempre que possível.

Cubra:

* Quando usar TaaC.
* Diferença entre:

  * unit test;
  * integration test;
  * component test;
  * contract test;
  * end-to-end test;
  * smoke test;
  * data quality test.
* Estratégia recomendada:

  * preferir testes autocontidos;
  * usar containers quando possível;
  * usar Testcontainers para dependências reais em container;
  * usar LocalStack/moto para AWS quando fizer sentido;
  * usar WireMock/mock server para APIs externas;
  * usar banco real em container quando possível;
  * evitar depender de ambiente compartilhado instável;
  * evitar teste integrado que vira E2E lento e frágil.
* BDD:

  * Avalie se faz sentido.
  * Recomende BDD apenas quando a linguagem Gherkin realmente ajudar negócio/QA/devs a alinhar regras.
  * Não usar BDD como padrão para todo teste técnico.
  * Para testes técnicos, prefira pytest/JUnit/Testcontainers com nomes claros.
  * Para fluxos de negócio críticos e legíveis por áreas não técnicas, BDD pode ser usado com Cucumber/behave/pytest-bdd.
* Test data management.
* Isolamento.
* Cleanup.
* Paralelismo.
* Determinismo.
* Tempo máximo aceitável.
* Como rodar local.
* Como rodar na pipeline.
* Como debugar.
* Como versionar fixtures.
* Como validar S3/DynamoDB/SQS/SNS/API/DB quando aplicável.
* Como validar pipeline Airflow/dbt/Glue em níveis diferentes.
* Critérios para não exagerar no E2E.
* Estratégia de pirâmide de testes adaptada para data engineering.

Inclua exemplos de:

* teste integrado Java com Testcontainers.
* teste integrado Python com LocalStack ou alternativa.

## 11. Testes unitários

Crie `docs/padroes/08-testes-unitarios.md`.

Regras:

* Cobertura mínima obrigatória: 90%.
* Medir line coverage e branch coverage quando a ferramenta permitir.
* Exceções devem ser explícitas e justificadas:

  * DTO puro;
  * configuração trivial;
  * bootstrap;
  * código gerado.
* Teste deve validar comportamento, não implementação.
* Evitar teste que apenas aumenta cobertura sem assert relevante.
* Nomes de testes devem explicar cenário, ação e resultado.
* Padrão Given/When/Then pode ser usado sem framework BDD.
* Testes devem ser rápidos, determinísticos e independentes.
* Mocks devem ser usados para bordas; regra de negócio deve ser testada sem mocks excessivos.
* Para código de dados, testar transformação com datasets pequenos e representativos.
* Para Airflow, testar parse, estrutura e funções auxiliares.
* Para dbt, usar testes nativos, unit tests e fixtures.
* Para Terraform, usar fmt/validate/lint e testes de módulo quando aplicável.

Inclua exemplos para:

* Java Spring Boot com JUnit 5, Mockito e AssertJ.
* Python Lambda com pytest.
* Glue/PySpark com pytest.
* Airflow DAG parse test.
* dbt test.
* Terraform validate/lint.

## 12. Testes de mutação

Crie `docs/padroes/10-testes-de-mutacao.md`.

Regras:

* Mutation score mínimo: 90%.
* Aplicar prioritariamente em código de regra de negócio, transformação crítica, validação, cálculo, decisão e mapeamento.
* Evitar exigir mutation testing pesado em:

  * DTO puro;
  * configuração;
  * bootstrap;
  * scripts sem lógica;
  * Terraform;
  * SQL/dbt de forma literal, salvo estratégia específica.
* Para Java:

  * usar PIT/Pitest.
  * configurar threshold de 90%.
  * rodar inicialmente em módulos críticos para controlar tempo.
* Para Python:

  * usar mutmut ou ferramenta equivalente.
  * configurar paths de mutação.
  * excluir handlers finos, bootstrap e código sem lógica.
* Para Glue:

  * aplicar mutação nas funções puras de transformação, não necessariamente no job Spark inteiro.
* Para dbt:

  * explique que mutation testing tradicional não é tão direto.
  * recomende abordagem equivalente com testes negativos, fixtures, data quality tests e alteração controlada de fixtures para provar que os testes falham.
* Para Airflow:

  * aplicar mutação em funções auxiliares, não no DAG declarativo em si.
* Explique como interpretar mutantes sobreviventes.
* Explique como evitar “matar mutante” com teste frágil.
* Explique como configurar na pipeline.
* Explique estratégia incremental:

  * obrigatório em código novo;
  * legado pode entrar com baseline;
  * módulos críticos devem convergir para 90%.

Inclua exemplos de:

* configuração PIT Maven/Gradle.
* configuração mutmut.
* comando de pipeline.
* relatório esperado.

## 14. Performance

Crie `docs/padroes/12-performance.md`.

Performance deve ser preocupação transversal em implementação e code review.

Cubra:

* Pensar em volume, cardinalidade, concorrência e custo antes de implementar.

* Evitar N+1.
* Evitar chamadas em loop para rede/AWS/DB.
* Usar batch quando aplicável.
* Paginação.
* Streaming quando aplicável.
* Timeouts.
* Retries com backoff e jitter.
* Circuit breaker quando aplicável.
* Pool de conexão.
* Cache com critério.
* Idempotência e deduplicação.
* Evitar processamento desnecessário.
* Evitar serialização/deserialização excessiva.
* Evitar `select *`.
* Filtrar cedo.
* Particionar dados.
* Usar formato colunar para dados analíticos quando aplicável.
* Evitar `collect` em Spark.
* Evitar parse pesado de DAG.
* Otimizar Lambda:

  * memória;
  * CPU;
  * cold start;
  * package size;
  * reuso de client.
* Otimizar Java:

  * queries;
  * pool;
  * JVM;
  * serialização;
  * paginação.
* Otimizar Terraform:

  * módulos claros;
  * evitar dependências desnecessárias;
  * evitar recursos superdimensionados.
* Performance testing:

  * microbenchmark só quando necessário;
  * teste de carga para endpoints;
  * teste de volume para jobs;
  * teste com dataset representativo;
  * baseline antes/depois.
* Code review de performance:

  * checklist;
  * sinais de alerta;
  * perguntas obrigatórias.
